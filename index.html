Talk to your data.

Ask in plain language; get answers you can trust. Malloyyo serves your Malloy semantic model to any AI over MCP: measures, dimensions, and joins are defined once in the model, your AI composes queries against it instead of guessing at SQL, and the same question gets the same right answer every time. All you need is a database connection (or an S3 bucket) and a GitHub repository for the model.

This is what talking to your data looks like

Pick a dataset and click a ★ question: the question becomes a Malloy query composed against the model, the query compiles to SQL, and your answer comes back, live in this page. Then edit anything. That loop is the whole product.

How to talk to YOUR data

  1. Serve your model

    Describe your data once in Malloy: sources, joins, measures. Publish it with two commands (malloyyo login, then malloyyo publish), or point Malloyyo at a GitHub repo and a webhook keeps it fresh on every push. DuckDB is built in, so Parquet on S3 or any web server works with no warehouse at all.

  2. Connect your AI

    Add the /mcp endpoint to Claude (or any MCP client) and sign in. The AI discovers the tools it needs on its own: list_sources, describe_source, query, and yo_help. From that moment it knows what your data means, not just what tables exist.

  3. Ask in plain language

    Which manufacturers recalled the most vehicles? Where does revenue come from? The AI composes governed Malloy against your model, never raw SQL, so the numbers stop drifting. Every query is logged in the ltool browser, where you can edit, favorite, re-run, and share it.

order_items

Order line items from the example models set (ecommerce sample). ★ What are the top brands within each product category, ranked by category total sales? ★ Monthly sales trend. ★ Top 5 brands by total sales, with percent of sales, gross margin, and item count. ★ Which customers return a lot and what did they buy? ★ Among customers with at least 5 purchases, who has the highest return rate? ★ Which product categories have the best gross margin percentage vs raw sales volume? ★ Top 10 states by total sales. ★ (Chart) All product categories in the ecommerce store by order count.

run: order_items -> {
  group_by: product_category
  aggregate: category_sales is sum(line_total)
  nest: top_brands is {
    group_by: brand
    aggregate: sales is sum(line_total)
    order_by: sales desc
    limit: 3
  }
  order_by: category_sales desc
}

auto_recalls

Auto recalls from the NHTSA, analyzed by component category and recall trends over time. Data source: nhtsa.gov/recalls. ★ Which manufacturers have recalled the most vehicles in total (by number of vehicles affected)? ★ Which component categories account for the most recalls and affected vehicles? ★ Recall counts by year since 2000. ★ The largest Do Not Drive recalls by vehicles affected, and what fraction of owners actually got the fix (completion rate)? ★ Porsche recalls grouped by subject.

run: recalls -> {
  group_by: manufacturer
  aggregate:
    campaigns is count()
    vehicles is sum(vehicles_affected)
  order_by: vehicles desc
}

baby_names

Baby names from the Social Security Administration. ★ Which names from the 1960s were most over-represented in Louisiana compared to their national popularity? ★ Which names were most overrepresented in the 1950s relative to their share across all years? ★ Top names in each decade. ★ Find female names that were popular in southern states in the 1920s but are rarely used today. ★ Tell me about the name Kyle: popularity over time, gender breakdown, peak years, and top states.

run: names -> {
  group_by: decade
  nest: top_names is {
    group_by: name
    aggregate: births is sum(births)
    order_by: births desc
    limit: 3
  }
}

How it fits together

The architecture from the repository README: a thin server between your AI and your data, with the semantic model as the only doorway.

   MCP client (claude.ai)            Browser (you)

   |  MCP server /mcp |          |      Web UI       |

   +----------------------v-----------------------+       Authoring models

   +------+-----------------------------+---------+       · Claude (already knows Malloy)
          |                             |

   |  metadata   |             |  Analytical DB  |
   |  datasets   |             |  BigQuery ·     |
   |  models     |             |  Snowflake ·    |
   |  users      |             |  MotherDuck ·   |
   |             |             |  (or S3/GCS)    |

Publish in two commands: malloyyo login <target> then malloyyo publish <target> bundles your *.malloy files, records the git commit, and pushes. If the model does not compile, the push is rejected and the live model is untouched. Or point Malloyyo at a GitHub repo and a webhook refreshes it on every push. Install: npm i -g @malloydata/malloyyo (Node 20+).

Why Malloyyo?

A thin layer between the AI and your data. It serves the semantic model; the model keeps the numbers honest.

Consistent by construction

The same question gets the same right answer

AI plus a raw database means invented columns, wrong joins, and fan-out double-counts that still look right. With a Malloy semantic layer in between, the AI composes queries against measures and joins that were defined once, correctly. The numbers stop drifting.

Tight control

The model is the boundary

The AI can only query what the semantic model exposes. Nothing outside it is reachable. Givens parameterize row-level restrictions, and a restricted query mode shuts off imports and raw table access entirely.

Develop, then publish

One command from repo to live endpoint

Build the model locally, then publish it with the CLI, or point Malloyyo at a GitHub repo and a webhook refreshes it on every push. If the model does not compile, the push is rejected and the live model is untouched. DuckDB is built in, so Parquet over plain HTTP works with no warehouse at all.

Start talking to your data.

One-click deploy to Vercel, or self-host with Docker. Claude already knows Malloy, so authoring the model is fast: describe your data once, and every question after that is a conversation.
